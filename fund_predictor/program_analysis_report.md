# 程序问题分析与优化方案

## 结论概览

这个项目的主体流程是完整的：数据抓取、特征构造、模型选择、训练、预测和结果持久化都已经串起来了。当前最值得优先处理的问题，不是语法错误，而是几个会直接影响结果可信度和运行稳定性的逻辑问题：数据刷新语义不一致、模型监控文件路径不统一、以及训练阶段把测试集同时用于选模和回测评估。

## 发现的问题

### 1. 指数数据的强制刷新参数没有真正生效

在 [backend/app/services/data_service.py](backend/app/services/data_service.py#L234) 中，get_index_daily() 只要本地缓存未过期就直接返回缓存，即使调用方传入了 require_fresh=True。这意味着“强制刷新”对指数数据不起作用，调用者会误以为拿到了最新行情，但实际仍然在用旧缓存。

影响：

1. 训练任务在要求最新数据时可能仍然使用旧指数，导致特征和标签口径不一致。
2. 线上预测在数据刚更新后，仍可能沿用过期缓存，降低结果时效性。

### 2. 模型监控文件使用了相对路径，和主模型目录不一致

模型归档目录是通过 [backend/app/services/model_registry_service.py](backend/app/services/model_registry_service.py#L15) 中的 MODEL_DIR 统一管理的，但模型监控文件在 [backend/app/services/prediction_service.py](backend/app/services/prediction_service.py#L471) 里却写死成了相对路径 models/.../model_monitoring.json。

影响：

1. 如果程序不是从项目根目录启动，监控文件会写到错误位置，或直接读不到历史监控记录。
2. 同一套模型元数据会被分散到两个路径体系里，后续维护和备份都会变复杂。

### 3. 训练流程把测试集既用于选模，又用于回测指标生成

在 [backend/app/services/model_selection_service.py](backend/app/services/model_selection_service.py#L564) 开始的流程里，模型先在测试集上计算最终分数并选出最佳候选，然后同一个测试集又被拿来生成 backtest、方向信号和覆盖率等结果。这样会让报告里的指标偏乐观，因为测试集已经参与了模型选择。

影响：

1. 回测指标不再是独立的泛化评估，而更像是调参过程的一部分。
2. 业务侧看到的 RMSE、方向胜率、区间覆盖率等指标会偏高，不利于真实判断模型质量。

### 4. 异常分层还可以再细化

目前应用层对所有 AppError 基本都返回 400，见 [backend/app/main.py](backend/app/main.py#L24) 和 [backend/app/main.py](backend/app/main.py#L30)。这会把数据源故障、外部接口超时、模型缺失、参数错误混在同一类 HTTP 语义里。

影响：

1. 前端和调用方无法根据状态码快速判断是用户输入问题还是系统问题。
2. 监控和告警也不容易区分“可重试的外部故障”和“需要用户修正的请求错误”。

## 优化方案

### 优先级 P0：先修会影响结果可信度的问题

1. 修正指数缓存逻辑，让 require_fresh=True 时无条件走远端刷新或至少先检查缓存是否符合刷新语义。
2. 把模型监控文件路径改成和 MODEL_DIR 一致的绝对路径，避免不同启动目录造成路径漂移。
3. 重构训练评估流程，把最终测试集留给一次性回测，模型选择只使用训练集和验证集，必要时再加一层独立验证切分。

### 优先级 P1：提高可维护性和可观测性

1. 为数据源、模型归档、监控文件统一提供路径构造函数，禁止在业务代码里直接拼接相对路径。
2. 把训练评估过程拆成更明确的阶段输出，例如训练、验证、最终回测、保存归档，减少一个函数承担过多职责。
3. 给训练流程补充最小化回归测试，重点覆盖缓存刷新、模型路径和测试集隔离三类问题。

### 优先级 P2：提高工程健壮性

1. 对外部数据源增加更清晰的失败分级，例如区分网络超时、数据为空、字段缺失、内容过期。
2. 给 API 错误码和 HTTP 状态码建立更明确的映射，至少区分 400、404、409、422、503 这几类常见场景。
3. 把训练和预测中复用的路径、阈值、窗口参数进一步集中到配置层，减少散落在服务代码中的硬编码。

## 建议的落地顺序

1. 先修数据刷新和路径一致性，因为这两项改动小、风险低，但能立刻提升行为正确性。
2. 再改训练评估拆分，因为这会影响指标口径，需要同步调整报告和前端展示。
3. 最后补测试和错误分层，让后续迭代更容易定位问题。

## 备注

本次快速检查没有发现语法错误，当前问题主要集中在运行语义和评估方法上，而不是代码能否启动。
