### 当前系统存在的问题和优化建议

---

#### 性能问题
1. **数据获取完全串行** — `load_market_data()` 中指数、基金、股票逐个下载，应改用 `ThreadPoolExecutor` 并发获取
2. **特征工程无缓存** — 每次训练都从原始CSV重新构建特征矩阵，对高频更新场景效率低

#### 内存泄漏风险
3. **模型对象常驻内存** — `prediction_service.py` 中 `joblib.load` 加载的模型在请求间共享但无过期清理机制
4. **DataFrame未及时释放** — 大规模特征矩阵在内存中停留时间过长

#### 数据库问题
5. **SQLite 并发瓶颈** — 多用户同时触发训练时可能锁表；建议加 WAL 模式或迁移 PostgreSQL
6. **无连接池** — 每次 API 请求创建新数据库连接

#### 代码质量
7. **魔法数字散落** — `MIN_TRAIN_ROWS=220`, `TRAIN_WINDOW=180`, `n_estimators=120` 等应集中到 config.yaml
8. **异常处理粒度不均** — 部分 akshare 调用裸 except 吞掉错误
9. **缺少类型注解** — 大量函数参数和返回值无 type hint

#### 用户体验
10. **前端技术栈陈旧** — 纯 HTML+JS，无组件化，难以维护扩展
11. **无多基金对比视图**
12. **预测结果无可视化图表**

#### 功能增强建议
13. **引入 Celery 异步任务队列** — 替代 BackgroundTasks，支持任务优先级和重试
14. **引入 Optuna 超参优化** — 当前模型参数全部硬编码

## 🐛 程序存在的问题与优化方案

### 1. 性能问题
- **宏观数据获取阻塞**: `macro_features.py` 中每次调用 `build_macro_features()` 都会重新请求 akshare，虽然有6小时缓存但进程重启后丢失。建议改用定时任务后台刷新 + Redis共享缓存
- **FOF递归深度风险**: `fof_service.py` 的 `MAX_FOF_RECURSION_DEPTH=3` 在极端情况下可能导致大量串行API调用。建议对嵌套FOF增加超时控制和并行化处理
- **特征缓存 pickle 安全性**: `feature_cache.py` 使用 `pd.read_pickle()` 存在反序列化安全风险。生产环境应使用更安全的格式（如 parquet/feather）
- **融合权重计算开销**: `_compute_fusion_weight()` 每次估算都要回溯30日历史做误差计算，CPU密集。建议将权重预计算并定期更新

### 2. 内存泄漏
- **内存缓存无上限**: `feature_cache.py` 的 `_memory_cache` 字典无LRU淘汰机制，长时间运行可能累积过多缓存条目。应添加最大容量限制或使用 `functools.lru_cache`
- **盘中估算缓存无限增长**: `intraday_service.py` 的 `_intraday_cache` 全局字典只增不减，多基金场景下会持续膨胀。需添加过期清理机制
- **任务队列引用残留**: `task_queue.py` 中失败的任务信息永久保存在 `_tasks` 字典中，长期运行可能导致内存增长。应添加定期清理已完成任务的逻辑

### 3. 数据库问题
- **冷启动模型无持久化**: `cold_start.py` 的群体模型预测结果未入库记录，无法追溯历史预测质量。建议增加 prediction_history 表记录
- **FOF预测结果无存储**: `fof_service.py` 的递归预测结果仅返回未持久化，不利于后续分析和监控

### 4. 代码质量问题
- **`set_log_context` 未导入**: `macro_features.py` 和部分文件使用了 `set_log_context()` 但未在文件顶部导入（依赖运行时可用）。应在各模块统一导入
- **类型注解不完整**: 部分函数缺少返回值类型注解，影响IDE支持和静态检查
- **硬编码魔法数字**: 如 `MIN_DAYS_FOR_COLD_START=120`, `FUSION_WINDOW=30`, `REGRESSION_WINDOW=20` 等散落在各文件中，应统一收归 `config.py`
- **异常处理粒度不一致**: 部分地方用宽泛的 `except Exception` 吞掉了具体错误类型，不利于问题定位

### 5. 用户体验问题
- **盘中估算延迟感知**: POST 触发估算后用户不知道进度，建议增加 WebSocket 推送或轮询状态端点
- **错误提示不友好**: API 返回的技术性错误信息（如 akshare 数据源异常）对前端用户不够友好，需要增加用户可读的错误映射层
- **FOF预测耗时不可控**: 嵌套FOF可能涉及数十个子基金预测，响应时间可能很长。应增加超时设置和异步模式

### 6. 功能增加建议
- **实时数据推送**: 盘中估算可结合 WebSocket 实时推送净值变化曲线
- **A/B测试框架**: 融合权重可引入在线学习动态调整，而非仅基于历史误差
- **模型热加载**: hyperopt_service 优化出的参数应能自动替换当前模型配置
- **指标监控面板**: 为新增模块添加 Prometheus 指标导出（缓存命中率、队列深度、估算延迟等）
- **配置中心**: 将分散在各模块的超参数（窗口大小、阈值、TTL等）统一到 config.yaml 管理

### 当前仍存在的问题和优化建议

#### 性能问题
1. **宏观数据获取阻塞** — 每次调用都请求 akshare，建议定时任务后台刷新 + Redis 共享缓存
2. **FOF递归深度风险** — 极端FOF可能导致大量串行API调用，需加超时控制和并行化
3. **特征缓存 pickle 安全性** — 生产环境应改用 parquet/feather 格式
4. **融合权重计算开销** — 每次盘中估算都要回溯30日历史，应预计算并定期更新

#### 内存泄漏风险
5. **内存缓存无上限** — `_memory_cache` 字典无 LRU 淘汰，长时间运行可能膨胀
6. **盘中估算缓存无限增长** — `_intraday_cache` 只增不减，需添加过期清理
7. **任务队列引用残留** — 失败任务永久保存在 `_tasks` 中，需定期清理已完成任务

#### 数据库问题
8. **冷启动/FOF预测结果无持久化** — 不利于后续分析和监控，建议增加 prediction_history 表

#### 代码质量
9. **部分文件 set_log_context 未导入** — macro_features.py 等新模块需统一 import
10. **魔法数字仍有散落** — 如 `MIN_DAYS_FOR_COLD_START=120`, `FUSION_WINDOW=30`, `REGRESSION_WINDOW=20` 等，应全部收归 config.yaml
11. **异常处理粒度不一致** — 部分宽泛 except 吞掉具体错误类型

#### 用户体验
12. **盘中估算延迟感知** — POST 触发后无进度反馈，建议 WebSocket 推送或轮询状态
13. **错误提示不友好** — 技术性错误信息对前端用户不够友好，需增加用户可读映射层
14. **FOF预测耗时不可控** — 可能涉及数十个子基金预测，需超时设置和异步模式

#### 功能增强建议
15. **WebSocket 实时推送** — 盘中估算、训练进度可通过 WebSocket 推送替代轮询
16. **A/B测试框架** — 融合权重可引入在线学习动态调整
17. **模型热加载** — hyperopt 优化参数应能自动替换当前配置
18. **Prometheus 指标导出** — 缓存命中率、队列深度、估算延迟等监控指标
19. **配置中心热更新** — config.yaml 变更无需重启即可生效
20. **前端 PWA 支持** — Service Worker 实现离线访问和桌面安装
