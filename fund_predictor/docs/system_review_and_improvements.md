## 一、严重缺陷（优先修复）

> 这类问题会直接导致运行报错或产生错误的预测结果，应在下一个版本中优先解决。

---

### 1.1 冷启动盲区：120–220 天基金无处理路径

**问题根源**

冷启动触发条件是 `history_days < 120`，而正常训练流程的最低要求是 `MIN_TRAIN_ROWS = 220`。当一只基金历史天数在 120–220 之间时，冷启动逻辑判断 `blend_weight >= 1.0`，直接 `RETURN 0.0`，系统随即进入正常训练流程——但数据不足 220 行，立即抛出 `InsufficientDataError`。这个区间的基金处于"两边都不管"的真空地带，任何对其发起预测的请求都会以报错告终。

**影响范围**

所有成立 4–10 个月的基金（新发基金高峰期大量存在）。

**修复方向**

将冷启动阈值与 `MIN_TRAIN_ROWS` 对齐，冷启动的过渡期应延伸到数据满足最小训练要求之后才开始收敛。具体方案：

- 0–119 天：100% 群体模型；
- 120–219 天：仍使用群体模型（blend_weight 固定为 1.0），不触发个体训练；
- 220 天以上：开始 blend_weight 从 1.0 线性衰减至 0（在 400 天左右完全切换到个体模型）。

同时将 `cold_start.group_model_days` 配置从 90 天提升至至少与 `MIN_TRAIN_ROWS` 一致（推荐 250 天，留一定余量）。

---

### 1.2 WF-CV 参数与最小样本量严重失配

**问题根源**

`walk_forward_train_months = 24`（约 500 个交易日），但 `min_train_rows = 220`（约 1 年）。对于刚满足最小样本要求的基金，其全部数据量远不够填满一个 WF-CV 训练窗口，`min_rounds = 12` 的要求也无法达到（12 轮需要约 38 个月 = 3 年历史数据）。结果是 WF-CV 静默跑了极少轮次（甚至 0 轮），验证结果毫无统计意义，却不会产生任何报警。

**影响范围**

历史数据不足 3 年的基金（这在实践中占绝大多数，尤其是近 5 年发行的新基金）。

**修复方向**

在训练入口处根据实际数据量动态计算 WF-CV 参数，而非使用固定配置：

- 计算可用训练数据总长度 `T`；
- 动态设 `train_window = min(500, T × 0.55)`，`valid_window = min(60, T × 0.22)`；
- 动态计算最多可运行轮数，若低于 3 轮则降级为简单 Hold-out 验证（最后 20% 作为验证集）并在结果中标注"数据不足，使用简化验证"；
- 在 `metrics.json` 中记录实际使用的轮数，前端展示时对轮数 < 5 的模型用明显标识提示预测置信度较低。

---

### 1.3 保形预测校准集来源未明确（潜在数据泄露）

**问题根源**

Step 5 的输入是"已训练 model, 校准集 (X_calib, y_calib)"，但全文档没有说明校准集对应四段划分中的哪一段。最大的嫌疑是使用了 Valid 集（22%）——但 Valid 集已经用于模型选择竞争和 Stacking 元学习器训练。将同一批数据再次用于保形预测校准，会系统性低估真实残差（因为模型在这批数据上已经过拟合了一些规律），导致置信区间偏窄，实际覆盖率低于声称的 90%。这意味着用户看到的"90% 置信区间"实际上可能只有 80% 甚至更低的覆盖率。

**修复方向**

明确指定用 `Test_select`（13% 段）作为保形预测的校准集，该段未参与任何训练步骤，校准结果才是真正无偏的。在代码中通过命名约定强制约束（如将该段命名为 `X_calib`），并在技术文档和代码注释中同步标注这一约定，防止后续维护者误操作。

### 1.4 Benchmark 映射逻辑有误

**问题根源**

Level 1 分类将含"沪深300×80%+中证全债×20%"的基准映射为 `hybrid_balanced`。但 80% 的股票权重明确表示这是偏股型基金，混合均衡型通常指股票仓位在 40–60% 之间（对应基准约为"沪深300×50%+中债×50%"）。这一映射错误会导致大量主动偏股型基金被路由到平衡混合流程，使用股债各半的特征集，因子配置与实际持仓结构严重错位，预测误差会系统性偏大。

**影响范围**

所有设置了股票基准权重 > 65% 但仍用了"股债复合基准"写法的基金，这在中国主动偏股混合基金中相当普遍。

**修复方向**

解析基准字符串中各资产类别的数值权重：

- 股票权重 > 65% → `hybrid_equity`
- 股票权重在 40–65% 之间 → `hybrid_balanced`
- 股票权重 < 40% → `hybrid_bond`

对无法解析出权重数字的基准（如纯文字描述），回退到 Level 2 名称关键词推断，而非直接默认为 `hybrid_balanced`。

---

### 1.5 缓存策略与数据更新机制缺失

**问题根源**

`cache_stale_days = 10` 表明内存缓存可以沿用 10 天前加载的数据。但更根本的问题是：系统依赖的本地 CSV 文件由谁来更新？整份文档没有任何地方描述数据更新调度（无 cron、APScheduler、Celery Beat 等）。这意味着如果没有人工触发，本地 CSV 永远不会刷新，系统将持续用历史数据进行预测，且不会产生任何报警或提示。

**影响范围**

生产环境下的所有基金预测。一旦部署后无人维护，7 天后的预测结果已经参考的是过时数据。

**修复方向**

增加独立的数据更新调度模块：

- 每日 17:30（收盘后）自动触发 AKShare 拉取当日净值和市场行情，增量更新 CSV；
- 将数据最后更新时间写入 SQLite 的 `data_status` 表；
- 在预测接口中检查"数据距今是否超过 2 个交易日"，超过则在响应中附加 `data_warning` 字段提示前端展示警告；
- Dashboard 的 `AdminDataStatus` 页面展示每只基金和每个指数的数据新鲜度，标红超期项。

---

### 1.6 货币基金双重判断存在漏判漏洞

**问题根源**

路由层对货币基金有两个独立的判断条件：`skip_prediction == True` 和 `fund_type == "money_market"`，但 `fund_profile_service` 的分类逻辑并不保证两者始终一致。如果某只货币基金在分类时只触发了 `fund_type == "money_market"` 但 `skip_prediction` 未被正确设为 `True`（例如边界条件下的名称关键词识别），它会绕过第一个判断，进入最后的"其他"分支，被当作普通基金走完整 ML 流程，产生毫无意义的"预测净值"输出。

**修复方向**

统一判断逻辑，在 `classify_fund` 中保证以下约束：

```
IF fund_type == "money_market" THEN skip_prediction = True（强制，不可例外）
```

同时移除路由层中的 `fund_type == "money_market"` 冗余分支，所有跳过逻辑统一走 `skip_prediction == True` 一个入口，消除不一致性来源。

## 二、中等问题（影响预测质量）

> 这类问题不会导致系统崩溃，但会使预测精度、置信度或模型稳定性低于应有水平。

---

### 2.1 IC 计算使用 Pearson 而非 Spearman

**问题**

因子预筛选 Round 1 使用 `pearsonr` 计算信息系数。基金日收益率序列存在明显的厚尾分布，极端行情日（如熔断、大幅反弹）的涨跌幅远超正态假设，Pearson 相关系数对这类异常值高度敏感，可能因少数极端样本就大幅高估或低估因子有效性。业界量化因子评价的标准是 Spearman 秩相关（RankIC），其排序不变性对异常值天然鲁棒。

**修复方向**

将 IC 计算底层函数替换为 `spearmanr`，对应 `ic_threshold` 可同步调整至 0.03（Spearman IC 绝对值通常略低于 Pearson IC）。ICIR 的计算也应基于 Spearman IC 序列，而非当前的 Pearson IC 序列。

---

### 2.2 模型选择标准只看 MAE，忽略方向准确率

**问题**

当前选模型逻辑 `best_model = min(models, key=lambda m: m.valid_mae)` 纯粹以误差最小为标准。在实际使用场景中，用户更关心"明天是涨还是跌"这一方向性判断，而不是精确到小数点后几位的预测误差。一个预测值始终趋向于 0（永远预测不涨不跌）的模型 MAE 可能不大，但方向准确率等于 50%，与随机猜测无异。

**修复方向**

在 WF-CV 的每轮验证中同时记录方向准确率，并在模型选择中引入联合指标。推荐方案：

- 联合得分 `score = 0.6 × normalized_mae_rank + 0.4 × (1 - direction_accuracy_rank)`，两个指标分别标准化后加权；
- 或者设定方向准确率的最低门槛（如 > 52%），未达标的候选模型直接排除出竞选；
- 在 `metrics.json` 中额外输出 `direction_accuracy` 字段，前端在模型信息区展示。

---

### 2.3 VIF 逐步剔除存在路径依赖

**问题**

当前逻辑每次找出 VIF 最高的特征将其剔除，然后重算所有剩余特征的 VIF，循环直到满足阈值。问题在于：当多个特征的 VIF 接近时，剔除顺序具有随机性，不同批次运行可能保留不同的特征子集，导致模型的可复现性差，且无法保证保留下来的一定是"最有预测价值"的那个。

**修复方向**

改为基于相关性聚类的去重方法：

1. 计算所有特征两两之间的 Spearman 相关系数矩阵；
2. 将相关系数绝对值 > 0.80 的特征归为一个相关簇；
3. 每个相关簇只保留 IC 绝对值最大的代表特征；
4. 不同簇之间不进行跨簇删减。

这种方式的结果具有确定性（由 IC 大小决定），且明确保证了保留"最有预测价值"的特征。

---

### 2.4 `index_equity` 的规则引擎 TODO 未完成

**问题**

路由中注释了 `TODO: 规则引擎替换`，说明指数基金目前仍走完整 ML 流程。指数基金净值变动约等于"标的指数涨跌幅 − 日费用率"，这是一个已知的线性公式，规则精度理论上可达 99% 以上。让它走完整的 WF-CV + Stacking 流程不仅浪费计算资源（每次预测要花几十秒），模型反而要从噪声中反推一个已知的线性关系，精度还不如直接套公式。

**修复方向**

补全规则引擎，`index_equity` 类型完全绕过 ML 流程：

- 点预测：`pred_return = index_return × (1 - daily_fee_rate)`；
- 置信区间：使用历史跟踪误差的 ±2σ 作为固定区间（通常在 ±0.05% 以内）；
- T 日盘中估算：直接用标的指数实时涨跌幅，精度极高；
- 对折溢价率较大的 ETF，叠加一个折溢价回归修正项。

---

### 2.5 四段划分与 WF-CV 的边界关系未厘清

**问题**

文档同时描述了静态四段划分（Train 55% / Valid 22% / Test_sel 13% / Test_fin 10%）和动态 WF-CV，但没有说明 WF-CV 在哪个数据子集上运行。如果 WF-CV 覆盖了 `Test_final`（最后 10%），这段数据既被用于某些 WF-CV 轮次的训练，又被用于"泛化性能报告"，最终报告的 MAE 会系统性乐观（泄露未来信息）。

**修复方向**

明确约定：WF-CV 只在 Train + Valid 段（共 77%）上运行，`Test_select`（13%）用于保形预测校准，`Test_final`（10%）严格隔离用于最终泛化性能评估，任何训练和校准步骤均不触碰这段数据。在代码中通过明确的数据切割变量名（如 `X_wfcv`、`X_calib`、`X_final`）强制约束。

---

### 2.6 残差自适应修正的数据来源缺失文档

**问题**

Layer 3 的残差修正 `bias = mean(recent_errors[-10:])` 中，`recent_errors` 来自何处完全未说明。这是一个隐藏依赖——它需要一张存储历史预测误差的数据库表，而 `db/models.py` 的 ORM 定义中是否包含这张表、字段结构如何、首次预测时如何处理空列表，均无文档。若 `recent_errors` 在初次部署时为空，这段逻辑会静默地 `bias = 0`（无修正），还是抛出异常，行为不确定。

**修复方向**

在数据库中明确定义 `prediction_history` 表，记录每次预测的预测值和实际值。残差修正模块显式读取该表，并对以下边界条件做出明确处理：历史记录不足 5 条时跳过残差修正（避免少量样本的 bias 引入更大噪声），并在 `metrics.json` 中记录当前残差修正状态。

## 三、架构优化建议

> 不影响当前功能，但对系统的可维护性、稳定性和生产化程度有重要提升。

---

### 3.1 `_NAV_CACHE` 的线程安全问题

模块级全局 `dict`（`_NAV_CACHE`）在 FastAPI 多 worker 或多线程场景下进行并发读写没有锁保护。两个请求同时写入同一 key 可能产生竞态条件，导致缓存数据损坏或部分写入。

**修复方向**：使用 `cachetools.TTLCache` 配合 `threading.RLock` 替换，或者在单进程模式下使用 `functools.lru_cache` + 自定义 TTL 装饰器。如果迁移到多进程部署（多个 Uvicorn worker），则应改用进程外缓存（Redis）。

---

### 3.2 模型文件缺乏版本管理

`models/{fund_code}/model.pkl` 在每次重训后直接覆盖，没有保留历史版本。一旦新模型训练质量下降（常见于市场 regime 切换期），无法快速回滚，只能重新训练。

**修复方向**：按时间戳命名模型文件（如 `model_20260524.pkl`），同时维护 `latest.json` 记录当前激活的版本。保留最近 3 个版本（超出则删除最旧的），版本元数据（训练日期、验证 MAE、方向准确率）写入 `versions.json`，管理页面提供一键回滚功能。

---

### 3.3 同步预测链路的超时风险

`request_timeout = 30s`，而文档性能表中 WF-CV 单次耗时就可能达到 5–30s，加上数据获取和特征构建，完整链路时间可能突破超时限制。特别是首次预测一只基金时（需要先触发训练），前端会收到 504 而非有意义的错误信息。

**修复方向**：将"训练 + 预测"拆为两个独立步骤——训练走已有的异步任务队列，预测依赖已存在的 `model.pkl`（单次推理 < 50ms，完全不超时）。对无模型的基金，预测接口返回 `MODEL_NOT_FOUND`（HTTP 404），附加提示"请先在训练页面训练该基金"，而非在请求上下文内尝试训练。

---

### 3.4 缺少 API 认证与限流

当前所有 13 个 API 接口均为无认证状态。即使是内部工具系统，若部署在局域网或 Docker 暴露端口，也存在被滥用（如高频调用训练接口耗尽服务器资源）的风险。训练接口尤其敏感，单次调用可能消耗大量 CPU 和内存。

**修复方向**：最低配置为 API Key Header 认证（在 `config.yaml` 中设置密钥，中间件层校验 `X-API-Key` 请求头）。训练接口叠加频率限制（如每 IP 每分钟最多触发 2 次训练任务），防止意外的资源耗尽。

---

### 3.5 SQLite 在并发训练场景下的瓶颈

`max_concurrent_tasks = 3` 允许 3 个训练任务同时运行，叠加实时预测的读请求，SQLite WAL 模式在多写并发下仍会出现锁等待（WAL 只允许单个写入者）。如果 3 个训练任务同时写入结果，会造成明显的排队延迟。

**修复方向**：短期内将 `max_concurrent_tasks` 降为 2，降低竞争概率。中长期若并发需求增加，考虑迁移到 PostgreSQL（完整支持多写并发），并引入连接池管理（如 `asyncpg` + `SQLAlchemy async`）。

## 四、用户输入智能适配增强方案

> 这一章节专注于提升系统处理用户输入基金代码时的智能性与容错性，覆盖从输入清洗到结果解释的完整链路。

---

### 4.1 基金代码输入的智能容错与标准化

**当前问题**

系统直接将用户输入的 `fund_code` 传入后续流程，未做任何标准化处理。中国基金代码有以下常见的输入变体，直接传入 AKShare 会导致拉取失败并抛出 `DataFetchError`：

| 用户实际输入 | 问题 | 标准形式 |
|---|---|---|
| `"1100"` | 位数不足 | `"001100"` |
| `"110011 "` | 末尾空格 | `"110011"` |
| `"sh110011"` | 带交易所前缀 | `"110011"` |
| `"000001.OF"` | Wind/Bloomberg 格式 | `"000001"` |
| `"000001.SZ"` | 股票代码误输入 | 应提示用户输入基金代码 |
| `"沪深300"` | 输入的是名称而非代码 | 应触发名称搜索 |

**优化方案：输入标准化流水线**

在 `fund_profile_service.classify_fund` 的入口处，增加一个 `normalize_fund_code(raw_input)` 前置函数，按顺序执行以下步骤：

1. **去除空白字符**：strip 首尾空格、全角空格、不可见字符；
2. **去除常见后缀**：移除 `.OF`、`.SH`、`.SZ`、`.CSI` 等后缀；
3. **去除交易所前缀**：移除 `sh`、`sz`、`of`、`SH`、`SZ` 等前缀；
4. **纯数字补零**：若为纯数字且位数不足 6 位，左补零至 6 位（如 `"1100"` → `"001100"`）；
5. **格式合法性验证**：标准化后的代码应为 6 位纯数字，否则进入名称搜索路径；
6. **股票代码识别**：6 位数字但以 `"6"` 或 `"0"`/`"3"` 开头且不在基金代码范围内的，提示"该代码看起来是股票代码，请输入基金代码"（结合 AKShare 做类型验证）。

**名称搜索路径**：若输入为中文字符串或识别到是基金名称关键词（如"华夏成长""易方达"），调用 `ak.fund_name_em()` 模糊搜索，返回候选基金列表供用户选择，而非直接报错。

### 4.2 基金代码有效性快速预检

**当前问题**

系统在触发完整预测流程后才发现基金代码不存在，此时已消耗了数据拉取和特征构建的时间，用户等待后得到的只是一条报错。

**优化方案：三级快速预检**

在 `predict` 接口的最前端，用轻量级方式快速判断代码有效性，失败即早返回：

- **Level 1（本地黑名单检查，< 1ms）**：维护一份已知无效代码的本地缓存（如因清盘、合并而失效的代码），命中则立即返回"该基金已清盘/合并"并附上终止日期；
- **Level 2（本地历史数据检查，< 10ms）**：检查 `data/raw/fund_nav/{code}.csv` 是否存在，若存在则视为有效，跳过 Level 3；
- **Level 3（AKShare 轻量查询，1–3s）**：调用 `ak.fund_individual_basic_info_xq` 获取基金基本信息。若返回空或报错，则返回明确的"基金代码不存在"提示，并附上可能的相似代码建议（基于编辑距离搜索本地已知代码库）。

### 4.3 基金画像驱动的预检查与用户预期管理

**当前问题**

用户输入代码后直接触发预测，对结果可靠性没有任何预先提示。对于新基金、数据稀少的基金、已经停止申购的基金，系统应该在预测前主动告知用户潜在的局限性。

**优化方案：预测前画像摘要卡片**

在返回预测结果的同时（或在前端展示之前），附加一个 `fund_health_check` 对象，包含以下维度的评估：

```json
{
  "fund_health_check": {
    "data_sufficiency": "充足",          // 充足 / 偏少(<2年) / 不足(<1年)
    "data_freshness": "最新",            // 最新 / 轻微滞后(<2天) / 严重滞后(>2天)
    "model_age": "3天前训练",            // 提示用户模型是否需要重训
    "fund_status": "正常申购",           // 正常申购 / 暂停申购 / 限制大额申购
    "special_warnings": [
      "本基金处于季报披露窗口，预测置信度已下调15%",
      "基金经理于60天前变更，历史风格可能不再适用"
    ],
    "prediction_reliability": "中等"    // 高 / 中等 / 低
  }
}
```

前端根据 `prediction_reliability` 的值，在预测结果旁边用不同颜色的标识（绿/黄/红）直观展示可靠性等级，同时将 `special_warnings` 列表展示为可折叠的提示条，帮助用户自行判断是否依赖该预测结果。

### 4.4 相似基金推荐（结果不可用时的降级体验）

**当前问题**

当系统无法为某只基金提供预测时（代码不存在、数据不足、货币基金等），直接返回错误信息，用户无路可走。

**优化方案：不可用时推荐同类可用基金**

当预测无法执行时，附加 `similar_funds` 列表，推荐 3–5 只同类型、数据充足的基金作为参考：

- 推荐依据：同基金类型（`fund_type` 相同）+ 同基金公司（优先）或类似规模 + 预测数据充足 + 近期预测准确率较高；
- 对于货币基金，推荐页面直接展示 7 日年化收益率排行（从 AKShare 拉取），替代净值预测；
- 对于已清盘基金，展示其存续期间的表现摘要，并推荐同公司的在售同类基金。

### 4.5 基金类型自动校验与分类置信度

**当前问题**

`classify_fund` 以三级规则输出一个确定性的 `fund_type`，但规则分类存在错误概率（如前述 Benchmark 映射错误），且系统目前对分类结果没有置信度概念，错误分类会无声地传导到整个预测流程。

**优化方案：分类置信度评分 + 交叉验证机制**

在 `FundProfile` 结构中增加 `classification_confidence` 字段，基于以下规则打分：

| 条件 | 置信度 |
|---|---|
| Level 1 命中且基准权重可精确解析 | 高（0.9+） |
| Level 1 命中但权重无法解析（文字描述型基准） | 中（0.7） |
| Level 2 名称关键词命中 | 中（0.65） |
| Level 3 默认值 `hybrid_equity` | 低（0.5） |
| Level 1 结果与 Level 2 结果冲突 | 低（0.5），记录冲突日志 |

当置信度低于 0.6 时：

1. 系统在预测结果中标注"基金类型识别置信度较低，请人工确认"；
2. 在前端的基金画像页面，允许用户手动修正基金类型，修正值存入本地 `fund_type_overrides.json`，下次预测时直接使用；
3. 对大量置信度低的基金，定期导出供人工审核，用于持续改进分类规则。

---

### 4.6 预测结果的可解释性输出

**当前问题**

当前预测结果只有预测值、置信区间和方向概率，用户不知道这个预测主要由哪些因素驱动，难以判断结果是否可信。

**优化方案：SHAP 因子贡献度输出**

在预测流程的最后，使用 SHAP（Shapley Additive Explanations）计算各特征对当次预测的贡献度，并在响应中附加 `factor_contributions` 字段：

```json
{
  "factor_contributions": [
    { "factor": "fund_ret_lag1（昨日收益率）", "contribution": "+0.18%", "direction": "正向" },
    { "factor": "hs300_ret（沪深300昨日涨跌）", "contribution": "+0.12%", "direction": "正向" },
    { "factor": "beta_hs300（对大盘的敏感度）", "contribution": "+0.08%", "direction": "正向" },
    { "factor": "vol_ratio_20d（短期波动率）", "contribution": "-0.05%", "direction": "负向" }
  ],
  "top_driver": "昨日动量延续是本次预测看涨的主要驱动因子"
}
```

前端在预测结果页面中以水平条形图展示前 5 个正向/负向因子，配合自然语言摘要（如"本次预测主要依据昨日市场动量和当前大盘 Beta 暴露，驱动力较为集中，建议参考置信区间综合判断"）。这能显著提升用户对预测结果的理解和信任度，同时帮助快速识别异常预测（如某次预测的主驱动因子出现异常值）。

### 4.7 批量查询与历史回测对比的接口扩展

**当前问题**

系统目前每次只能预测一只基金（`POST /api/v1/fund/predict` 单只），如果用户想对比多只基金，需要多次请求，且没有历史预测准确率的快速查询入口。

**优化方案：批量预测接口 + 历史准确率摘要**

增加以下两个接口扩展：

- `POST /api/v1/fund/batch-predict`：接受最多 10 个基金代码列表，并发执行预测（利用已有的 `ThreadPoolExecutor`），返回所有基金的预测结果汇总，前端 `/compare` 页面直接对接；
- `GET /api/v1/fund/{code}/accuracy-summary`：返回该基金过去 30/60/90 日的预测准确率摘要（方向准确率、平均 MAE），帮助用户快速评估当前模型对该基金的历史预测质量，作为是否参考当次预测的参考依据。

### 4.8 数据质量驱动的自适应训练触发

**当前问题**

模型重训目前需要用户手动在 `/train` 页面触发，没有自动化的触发机制。在以下情况下，模型应该主动触发重训：

- 连续 5 日预测 MAE 超过历史均值 2σ（已在监控配置中定义，但缺少触发动作）；
- 距离上次训练超过 30 个交易日；
- 基金经理变更事件检测到。

**优化方案：智能重训触发器**

在每日数据更新调度模块（见 1.5 节）结束后，对所有已有模型执行一轮"健康检查"：

- 检查最近 5 日预测误差是否触发告警阈值；
- 检查模型最后训练时间是否超过 30 个交易日；
- 若触发任一条件，自动将该基金加入训练队列（优先级低于手动触发）；
- 在 Dashboard 的"模型状态"区域展示"待重训"列表，用户可一键确认批量重训。
